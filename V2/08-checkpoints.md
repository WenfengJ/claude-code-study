# 模块 8：Checkpoints（检查点）完全指南

> Level 1 | 预计学习时间：1小时 | Claude Code v2.1.101+

---

## 核心概念

Checkpoints 是会话状态的自动快照，包含：
- 所有对话消息历史
- 已做的文件修改
- 工具使用记录
- Session 上下文

**关键特性：**
- **完全自动**：每次用户提交提示词自动创建
- **持久化**：跨 session 保留（默认30天）
- **即时恢复**：恢复操作是瞬时的

---

## 访问检查点

```bash
# 方式1：键盘（最快）
Esc + Esc           # 打开检查点浏览器

# 方式2：命令
/rewind             # 打开检查点浏览器
/checkpoint         # /rewind 的别名
```

---

## 5个回滚选项详解

| 选项 | 代码文件 | 对话历史 | 适用场景 |
|------|---------|---------|---------|
| **恢复代码和对话** | ✅ 回滚 | ✅ 回滚 | 彻底重来 |
| **恢复对话** | ❌ 保留 | ✅ 回滚 | 保留代码，重试不同提示词 |
| **恢复代码** | ✅ 回滚 | ❌ 保留 | 撤销文件修改，保留对话上下文 |
| **从这里摘要** | ❌ 不变 | 压缩为摘要 | 释放上下文空间，继续工作 |
| **取消** | ❌ 不变 | ❌ 不变 | 取消操作 |

> 恢复后，被选节点的原始提示词会自动填回输入框，方便重新发送或修改。

## 核心使用场景

### 场景1：安全实验（最常用）

```
需求：尝试 Redis 缓存 vs 内存缓存，选最优方案

流程：
1. 当前状态 → [检查点 A 自动创建]
2. 实现方案 A（Redis缓存）→ 测试性能
3. Esc+Esc → 选"恢复代码和对话" → 回到检查点 A
4. 实现方案 B（内存缓存）→ 测试性能
5. 对比两种方案 → 选最优方案继续
```

### 场景2：大型重构保险

```
需求：重构认证模块，不确定影响范围

流程：
1. 重构前自动有检查点（每次提示词都创建）
2. 执行重构
3. 运行集成测试
4. 如果测试失败：Esc+Esc → 恢复到重构前 → 换策略
5. 如果测试通过：git commit 固化
```

### 场景3：上下文空间管理

```
问题：对话超长，Claude 质量开始下降

解决：
1. Esc+Esc 打开检查点
2. 选择一个节点 → "从这里摘要"
3. Claude 将该节点之后的内容压缩为摘要
4. 上下文空间释放，质量恢复
5. 继续工作

注意：摘要时文件不变，只有消息被压缩
```

### 场景4：A/B 设计对比

```
需求：对比两种数据库 Schema 设计

流程：
1. 初始状态 → 记录检查点位置
2. 实现 Schema A（用户表设计方案1）
3. Esc+Esc → 恢复到初始
4. 实现 Schema B（用户表设计方案2）
5. 两种 Schema 都实现完毕，评审选择
```

## 多分支实验工作流

```
初始状态（检查点 ROOT）
    │
    ├── 分支1：/branch redis-cache
    │     → 实现 Redis 方案
    │     → 测试通过
    │     → ✅ 选择这个方案
    │
    └── 分支2（从 ROOT 回滚后）
          → 实现内存缓存方案
          → 测试通过
          → 对比后放弃
```

**使用 `/branch` 配合 Checkpoints：**
```bash
/branch before-refactor    # 在会话中创建命名分支点
# 继续重构...
# 发现问题 → Esc+Esc → 恢复到 before-refactor 节点
```

## 与 Git 的协作关系

| 特性 | Git | Checkpoints |
|------|-----|-------------|
| 范围 | 文件系统 | 对话 + 文件 |
| 持久性 | 永久 | 30天 |
| 粒度 | 手动提交 | 每次用户提示自动 |
| 速度 | 需要 add/commit | 即时 |
| 共享 | ✅ push/pull | 有限 |
| 历史 | 清晰的 log | 时间线 |

**黄金组合：**
```
1. 用 Checkpoints 快速实验多种方案（秒级回滚）
2. 确定方案后 git commit（永久固化）
3. 大功能前：先 git branch，再开始
4. 阶段完成：git commit + 继续
```

## 上下文监控与检查点时机

### 何时应该重置上下文？

**信号：**
- Claude 的回答开始变得重复或质量下降
- 对话超过 50-100 轮
- 任务阶段发生重大转换（从设计阶段到实现阶段）
- `/context` 显示使用率超过 80%

**操作：**
```bash
/context     # 查看当前上下文使用情况

# 如果接近限制：
# 方案1：Esc+Esc → "从这里摘要" → 压缩释放空间
# 方案2：git commit 当前工作 → 开新 session
# 方案3：/branch 创建新分支继续
```

### cc-context-stats（推荐工具）

[cc-context-stats](https://github.com/luongnv89/cc-context-stats) 在状态栏实时显示上下文区域：

| 状态 | 颜色 | 含义 |
|------|------|------|
| **Plan** | 🟢 绿 | 安全，可以规划和编码 |
| **Code** | 🟡 黄 | 避免开始新计划 |
| **Dump** | 🟠 橙 | 尽快结束，准备重置 |

## 检查点的限制

```
⚠️ 不被追踪的变更：
- Bash 命令的副作用（rm、mv、mkdir 等文件系统操作）
- Claude Code 外部的变更（你直接用编辑器修改的文件）
- 数据库变更（INSERT、UPDATE、DELETE）
- 网络请求的副作用（API 调用、消息发送）
```

## 配置

```json
// .claude/settings.json 或 ~/.claude/settings.json
{
  "cleanupPeriodDays": 30    // 检查点保留天数（默认30天）
}
```

---

## 故障排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| 找不到预期检查点 | cleanupPeriodDays 太短 | 调高保留天数 |
| 恢复失败 | 文件被外部工具修改 | 先 `git status` 处理冲突 |
| 摘要质量差 | 摘要节点选择不当 | 选择一个完整任务节点作为摘要点 |
