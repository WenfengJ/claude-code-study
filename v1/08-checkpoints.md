# 模块 8：Checkpoints（检查点）

> Level 1 基础 | 预计学习时间：30分钟

---

## 核心概念

Checkpoints 是会话状态的快照，包含：
- 所有对话消息
- 文件修改记录
- 工具使用历史
- Session 上下文

**自动创建**：每次用户提交提示词时自动创建检查点，无需手动操作。

---

## 访问检查点

### 方法1：键盘快捷键
```
Esc + Esc（按两次 Esc）
```
打开检查点浏览界面，选择任意历史节点。

### 方法2：斜杠命令
```bash
/rewind       # 打开检查点浏览器
/checkpoint   # 同 /rewind 的别名
```

---

## 回滚选项（5个选项）

回滚时会看到以下选项：

| 选项 | 说明 |
|------|------|
| **恢复代码和对话** | 同时还原文件修改和消息历史 |
| **恢复对话** | 只回滚消息，保留当前代码文件 |
| **恢复代码** | 只还原文件修改，保留完整对话历史 |
| **从这里摘要** | 将此节点之后的对话压缩为摘要，释放上下文空间（文件不变） |
| **取消** | 不做任何改变，返回当前状态 |

> 恢复对话或摘要后，被选节点的原始提示词会自动填回输入框，方便重新发送或修改。

## 使用场景

### 场景1：探索多种方案

```
情境：不确定用 Redis 还是内存缓存

步骤：
1. 当前状态 → [检查点 A 自动创建]
2. 实现 Redis 缓存方案 → [检查点 B]
3. 按 Esc+Esc → 回滚到检查点 A
4. 实现内存缓存方案 → [检查点 C]
5. 对比两种方案的优劣
```

### 场景2：安全重构

```
情境：需要大幅重构认证模块

步骤：
1. 开始重构前 → [检查点自动存在]
2. 执行重构
3. 运行测试
4. 测试失败 → Esc+Esc 回到重构前
5. 尝试更保守的重构方式
```

### 场景3：A/B 设计对比

```
情境：设计两种不同的数据库 schema

步骤：
1. 初始状态 → 记住这个检查点
2. 实现 Schema A
3. Esc+Esc → 回到初始状态
4. 实现 Schema B
5. 选择更好的方案继续
```

### 场景4：上下文空间管理

```
情境：对话太长，Claude 质量开始下降

步骤：
1. Esc+Esc 打开检查点浏览器
2. 选择某个节点 → "从这里摘要"
3. Claude 将该节点之后的内容压缩为摘要
4. 上下文空间得到释放，质量恢复
```

## 与 Git 的关系

| 特性 | Git | Checkpoints |
|------|-----|-------------|
| 范围 | 文件系统 | 对话 + 文件 |
| 持久性 | 永久 | 30天（自动清理） |
| 粒度 | 手动提交 | 每次用户提示自动 |
| 速度 | 较慢 | 即时 |
| 共享 | ✅ | 有限 |

**配合使用策略：**
- 用 Checkpoints 快速实验多种方案
- 用 git commit 固化最终选定的变更
- 在执行 git 操作前先确认检查点存在

---

## 限制

- **Bash 命令的文件系统变更不被跟踪**（如 `rm`、`mv`、`cp`）
- **Claude Code 外部的变更不被跟踪**（编辑器、终端直接修改的文件）
- **不是版本控制的替代品**，用 git 做永久审计

---

## 上下文监控建议

随着对话增长，上下文窗口逐渐满载，模型质量会静默下降。

建议在以下情况使用 `/rewind` + "从这里摘要"：
- 感觉 Claude 的回答质量下降
- 对话超过50-100轮
- 任务阶段发生了重大转换
